# Wizard Duel - Central Server API Documentation

This document explains the architecture and API of the Wizard Duel central server (`server.py`), which handles match recordings and game data.

## Overview

The server is built using **FastAPI**, a modern, high-performance web framework for Python. It acts as a central repository for match logs (`.wzrd` files) generated by the game clients.

### Repository Structure
- **Game Clients**: (C++) Record match events locally and upload them via `MatchUploader`.
- **Web Server**: (Python/FastAPI) Receives uploads and organizes them by `match_id`.

---

## API Endpoints

### 1. Upload Match
`POST /upload_match`

This endpoint receives multipart form data from the game client at the end of a match.

#### Form Fields:
| Field | Type | Description |
|-------|------|-------------|
| `player1` | `string` | Name of the first player (Host). |
| `player2` | `string` | Name of the second player (Joiner). |
| `winner` | `string` | Name of the winning player. |
| `duration` | `integer` | Match duration in seconds. |
| `match_id` | `string` | A unique ID (timestamp) shared by both players to group recordings. |
| `match_file`| `file` | The actual `.wzrd` log file. |

#### Storage Logic:
When a file is received, the server:
1. Extracts the `match_id`.
2. Creates a directory at `recordings/<match_id>/`.
3. Saves the `.wzrd` file inside that directory.

## Building a "Proper" Central Server

To transform the current `server.py` into a full-scale game backend, you should consider these components:

### 1. Database Integration
Instead of just printing logs to the terminal, use a database (like **Postgres** or **MongoDB**) to store match metadata (who won, how long it lasted, etc.).
- **Tools**: SQLAlchemy, Tortoise ORM.

### 2. Match Analysis Service
Create a background task that parses the `.wzrd` CSV file after it is uploaded.
- **Goal**: Calculate advanced stats like "Accuracy %", "Most used spell", or "Highest burst damage".
- **Benefit**: Display these on a web dashboard for players.

### 3. Replay System (The "Killer Feature")
Build a web-based replay viewer using JavaScript (or Three.js if you want 3D).
- **How**: The web app reads the CSV from the server and "plays back" the events (positions, spells) on an HTML5 canvas.

### 4. Authentication & Profiles
Implement JWT (JSON Web Tokens) so players can log in.
- **Result**: `recordings/user_id/match_id/` structure.
- **Benefit**: Match history, friend lists, and global leaderboards.

## Example Usage (Python Client)
```python
import requests

files = {'match_file': open('duel_log.wzrd', 'rb')}
data = {
    'player1': 'Wizard_Alpha',
    'player2': 'Wizard_Beta',
    'winner': 'Wizard_Alpha',
    'duration': 120,
    'match_id': '20260425_120000'
}

response = requests.post('http://localhost:3000/upload_match', data=data, files=files)
print(response.json())
```

## How to Build Your Own Match Server

If you want to rewrite or expand the server from scratch, follow these key steps:

### 1. Choose a Framework
For Python, **FastAPI** is recommended because it handles asynchronous file uploads and provides automatic validation.
- **Alternatives**: Flask (simpler, but slower), Django (overkill for this).

### 2. Handle Multipart Form Data
Game clients usually send data as `multipart/form-data`. This allows sending both regular text (player names) and binary data (the log file) in a single request.
- In FastAPI, use `Form()` for text fields and `UploadFile` for the match log.

### 3. Implement the Logic
Your server should follow this workflow upon receiving a request:
1. **Validation**: Ensure all fields (`player1`, `match_id`, etc.) are present.
2. **Directory Management**: Use `os.makedirs(path, exist_ok=True)` to create folders dynamically based on the `match_id`.
3. **Stream Saving**: Instead of loading the whole file into RAM, use `shutil.copyfileobj()` to stream the data directly to the disk.

### 4. Grouping by Match ID
The core trick to organizing multi-player data is the `match_id`. 
- **The Concept**: The Host generates a unique string and shares it with all clients during the handshake.
- **The Result**: When the server receives files from different players with the same `match_id`, it simply places them in the same folder, making it easy to compare the two perspectives of the same match.

### 5. Deployment
Use a production-ready runner like **Uvicorn** or **Gunicorn** to host the app so it can handle multiple simultaneous uploads from different matches without blocking.
